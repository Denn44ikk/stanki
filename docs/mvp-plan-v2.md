# MVP Plan v2

## Цель

Собрать минимальный показываемый продукт, в котором:

- есть кнопка `Витрина`;
- пользователь выбирает несколько станков;
- станки добавляются в проект;
- станки автоматически раскладываются на поле без пересечений;
- станки можно двигать;
- станки можно вращать только на `0°` и `90°`;
- экспорты `PNG`, `PDF`, `DXF` работают;
- `DXF` берёт текущее положение станков из проекта;
- интерфейс выглядит как продукт, даже без умной расстановки.

## Scope MVP

Что входит:

- витрина станков;
- выбор нескольких станков;
- добавление в проект;
- простая автораскладка;
- drag and drop;
- rotation `0/90`;
- preview-формы станков;
- экспорт.

Что не входит:

- AI-расстановка;
- обучение на DXF;
- сложные правила примыкания;
- технологическая оптимизация;
- сложные ограничения по потокам и обслуживанию.

## Принцип визуализации

Для веба используем не полную DXF-геометрию, а отдельный `display/preview` слой.

Для каждого станка должны существовать 3 слоя:

1. `previewShape`
   Лёгкая форма для витрины и канваса.

2. `placementMeta`
   Ширина, длина, точка вставки, повороты.

3. `fullGeometry`
   Полная геометрия для DXF-экспорта.

Текущий вывод:

- сырые SVG из DXF слишком тяжёлые;
- грубый контур слишком бедный;
- для MVP нужен промежуточный слой: узнаваемый, но лёгкий.

## UX MVP

### Главный сценарий

1. пользователь открывает проект;
2. нажимает `Витрина`;
3. видит список станков;
4. выбирает несколько станков;
5. нажимает `Добавить в проект`;
6. станки автоматически появляются на поле;
7. двигает их;
8. поворачивает двойным кликом;
9. экспортирует результат.

### Требования к витрине

В витрине должны быть:

- имя станка;
- preview;
- выбор нескольких позиций;
- кнопка `Добавить в проект`.

Пока допустимо отображать названия как:

- `U44`
- `U47`
- `U50`

Позже их заменим на нормальные наименования.

## Правило простой автораскладки

Для MVP используем не AI, а deterministic layout:

- порядок берём по порядку выбора;
- или по фиксированному порядку в витрине;
- ставим слева направо;
- если строка переполнена, переносим вниз;
- используем минимальный зазор;
- не допускаем пересечений.

Это не “умная инженерная логика”.
Это стартовая раскладка, чтобы пользователь не ставил всё вручную с нуля.

## Rotation MVP

Поворот:

- только `0°` и `90°`;
- по двойному клику;
- rotation хранится в `placement.rotation`;
- DXF-экспорт обязан использовать это значение.

## Экспорты

### PNG

- экранная схема как есть.

### PDF

- дублирование экранной схемы.

### DXF

- полная геометрия станков;
- текущие координаты и rotation;
- всегда после сохранения текущего состояния проекта.

Если экспорт не использует текущее состояние, MVP считается сломанным.

## Этап 1. Зафиксировать модель данных станка для MVP

### Цель

Сделать так, чтобы у каждого станка был единый набор данных для витрины, поля и DXF.

### Что правим

Файл:

- [shared/domain/machine-visuals.ts](D:/Разработка/Дополнительные%20проекты/10.%20Расстановка%20станков/stanki/shared/domain/machine-visuals.ts)

### Что должно быть в этом файле

Для каждого станка:

- `id`
- `title`
- `previewUrl`
- `url` или `fullGeometryUrl`
- `fileName`
- `width`
- `height`
- `note`

### Решение

- `previewUrl` используется только в UI;
- `url/fileName` используются для DXF-экспорта;
- никакой UI не должен рендерить тяжёлый `url`, только `previewUrl`.

### Критерий готовности

- витрина, карточки и канвас используют только `previewUrl`;
- DXF использует только полную геометрию.

# Definition of Done

MVP считается готовым, если:

- пользователь открывает `Витрина`;
- выбирает несколько станков;
- нажимает `Добавить в проект`;
- станки автоматически раскладываются;
- станки выглядят узнаваемо;
- станки не лагают при 10 объектах;
- станки двигаются;
- станки вращаются на `0/90`;
- `PNG`, `PDF`, `DXF` работают;
- `DXF` повторяет текущее состояние проекта.
