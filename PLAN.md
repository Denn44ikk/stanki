## Goal

Собрать понятную full-stack систему для проектирования линии:
- ручной режим сборки состава и схемы;
- AI-режим разбора текста менеджера и подготовки правок;
- сохранение проектов;
- экспорт в DXF и PDF.

## Architecture / Structure

Текущая согласованная структура проекта:

```text
src/
  app/                 # рабочий frontend Vite/React
  legacy/              # предыдущий MVP и референсный код
  main.tsx             # клиентский entrypoint
  styles/              # глобальные стили
server/
  ai/                  # интеграции с LLM
  exports/             # серверные экспортеры
  http/                # Express entrypoints и маршруты
  persistence/         # SQLite-хранилище
shared/
  domain/              # схемы, каталог, геометрия, layout-engine
  exporters/           # общие CAD-экспортеры
tests/
  integration/         # API и end-to-end сценарии
  legacy/              # тесты старого MVP
  unit/                # модульные тесты новой доменной логики
docs/
  architecture.md      # краткое описание слоёв и AI-flow
public/
  dxf-machines/        # полные SVG-геометрии для экспорта
  dxf-machines-preview/# PNG-превью средней детализации для UI
scripts/
  generate-machine-previews.mjs
data/
  projects.sqlite      # локальная БД, по умолчанию вне git
PLAN.md
AGENTS.md
```

## Decisions

- AI не пишет координаты напрямую в проект: текст интерпретирует AI, координаты строит детерминированный layout-engine.
- Реальный LLM вызывается только на backend через OpenRouter.
- При отсутствии `OPENROUTER_API_KEY` система работает в fallback-режиме на детерминированных правилах и алиасах.
- Старый single-screen MVP сохраняется как `legacy`, чтобы не терять референсный код, но не участвует в основном приложении.
- UI-превью станков должны брать отдельные PNG из `public/dxf-machines-preview`, а полные SVG из `public/dxf-machines` остаются для экспорта и детальной геометрии.
- Для витрины и карточек допустимы растровые preview-изображения средней плотности, если они лучше передают форму станка и не тянут в браузер полную инженерную геометрию.
- Ошибки API не должны прятаться внутри ветки выбранного проекта: уведомления о сбоях создания и загрузки должны быть видны даже когда проект ещё не открыт.
- `loadProjects` не должен самовольно переключать текущий проект после успешного `create/save`; текущий экран управляется явным результатом пользовательского действия.
- В рабочем canvas помещения визуал станка должен заполнять инженерный footprint размещения; если SVG-геометрия имеет другой aspect ratio, UI подстраивается под footprint, а точная геометрия остаётся в DXF-экспорте.
- Основной layout страницы не должен получать горизонтальный scroll из-за zoom канваса: страница остаётся статичной по ширине, а overflow локализуется внутри окна помещения.
- Рабочий viewer помещения использует лёгкие preview-ассеты для интерактивности; инженерные SVG загружаются только в экспортных сценариях.
- Тяжёлые frontend-модули можно грузить лениво, если это не ломает основной workflow: `ProjectCanvas` и DXF-витрина не обязаны входить в стартовый bundle.
- Для показа заказчику текущую версию следует позиционировать как MVP-демо/предпроектный инструмент, а не production-систему: ключевой сценарий работает, но ещё нет ролей, админки, удаления проектов и промышленного data-flow.
